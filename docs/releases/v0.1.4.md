# Flowness v0.1.4 릴리스 노트

- 날짜: `2026-06-19`
- 범위: 하네스 레이아웃, workflow enforcement, init/commit 흐름, context lookup 최적화

## 요약

v0.1.4에서는 Flowness 하네스의 기본 저장 구조를 `.flowness/` 중심으로 정리하고, issue 생성과 workflow 진행을 더 강하게 제어하도록 바꿨다. 또한 init 단계에서 프로젝트 컨텍스트를 빠르게 재사용할 수 있도록 compact context 파일을 추가했다.

## 주요 변경

### `.flowness/` 레이아웃 전환

- 모든 Flowness 산출물은 `.flowness/` 아래로 이동한다.
- 대상 디렉터리: `config`, `issues`, `logs`, `workflows`, `rules`, `skills`, `scripts`, `templates`, `state`
- 루트 `AGENTS.md`는 허용하지만, Flowness 전용 파일은 루트에 새로 만들지 않는다.

### `.agent/.codex` 신규 생성 중단

- `flowness init`은 `.agent/`와 `.codex/`를 새로 만들지 않는다.
- 기존 `.agent/`가 있으면 읽기 호환은 유지하되, 마이그레이션 경고와 함께 새 구조 사용을 권장한다.

### 의미 기반 issue slug

- issue ID는 요청 의도를 분석해 만든다.
- 무의미한 `ISSUE-001-UNNAMED-*` 형식은 중단했다.
- 예시:
  - `ISSUE-001-COMMUNITY-MVP-PLAN`
  - `ISSUE-002-SIGNUP-LOGIN`
  - `ISSUE-003-BOARD-CRUD`

### `run/step/status/evidence:add`

- 사용자 진입점을 명령형 흐름으로 정리했다.
- `flowness run "<request>"`
- `flowness step --issue ISSUE-ID`
- `flowness status --issue ISSUE-ID`
- `flowness evidence:add ...`

### workflow guard 강화

- issue/state/evidence JSON을 직접 수정하는 경로를 차단했다.
- state 변경은 runner/CLI를 통해서만 진행한다.
- step skip은 막힌다.
- human gate는 명시적 승인 전까지 진행이 막힌다.
- close는 evidence review 없이는 완료되지 않는다.
- state/log 불일치가 감지되면 progression을 멈추고 복구 지침을 출력한다.

### git init 자동화

- `flowness init`은 현재 폴더가 git repo가 아니면 `git init`을 실행한다.
- 기존 사용자 파일은 확인 없이 덮어쓰지 않으며, 필요한 경우 안전 백업 또는 확인 절차를 따른다.

### commit step 추가

- 모든 기본 workflow에 최종 commit step을 추가했다.
- commit step은 다음을 확인한다.
  - `.flowness/rules/commit-policy.md`
  - `git status`
  - 최근 log/graph
  - diff stat
- 커밋 메시지는 저장소의 기존 히스토리와 일관되게 선택한다.
- 의도한 파일만 커밋하고, 결과를 hash/message/changed files 형태로 보고한다.
- commit이 불가능하면 사유를 기록하고 중단한다.

### compact context files 추가

- init 시 다음 파일을 생성한다.
  - `.flowness/project-profile.md`
  - `.flowness/context-index.json`
  - `.flowness/commands.json`
  - `.flowness/harness-manifest.json`
- 반복적인 문서 탐색을 줄이기 위한 빠른 조회용 파일이다.

## 호환성 주의

- 기존 `.agent/` 읽기 호환은 당분간 유지된다.
- 다만 새 파일 생성은 `.flowness/`로만 이뤄지므로, 혼용된 작업공간에서는 경로 기대치가 엇갈릴 수 있다.
- 오래된 `.agent` 구조를 계속 참조하는 자동화가 있다면 마이그레이션이 필요하다.

## 검증

- 이 릴리스 작업에서는 build/test/audit 경로를 통과하는 상태로 정리했다.
- 실제 배포 전에는 프로젝트별 init 결과와 workflow 실행 경로를 한 번 더 확인하는 편이 안전하다.
