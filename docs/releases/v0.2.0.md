# Flowness v0.2.0 Release Notes

- Date: `2026-06-21`
- Scope: backward-compatible user-facing commands, safer workspace upgrades, code-review routing, commit safeguards, and release checks

## Summary

Flowness v0.2.0 expands the command surface with compact navigation and audit commands, adds safer upgrade flows for existing projects, and hardens the review and commit paths without changing the overall workflow model.

## Added

- `flowness locate` for focused file-area lookup.
- `flowness test --summary` for compact test summaries.
- `flowness audit --changed` for changed-file audits.
- `flowness upgrade --dry-run` and `flowness upgrade --apply` for safer workspace upgrades.
- `flowness review:run` for the code-review workflow path.
- Release validation commands for docs and package checks.

## Changed

- Commit workflow safeguards now resolve the repository from changed files instead of the process cwd.
- Multi-repo and nested-repo changes are blocked by default.
- Evidence Review and recorded checks are required before commit for code changes.
- Generated AGENTS, navigation, context-index, command, and log artifacts stay compact and current-state focused.
- README, CLI README, changelog, release checklist, and versioned release notes now describe the same release surface.

## Fixed

- Current-release references now point to `v0.2.0`.
- Release checks now verify docs, tests, build output, and `npm pack --dry-run` together.

## Migration Notes

- Existing projects should use `flowness upgrade --dry-run` first.
- Then run `flowness upgrade --apply` after reviewing the plan.
- Do not rerun `flowness init` on an existing project.

## Verification

- `npm run build`
- `npm test`
- `npm run audit`
- `npm run release:check`
- `npm pack --dry-run --workspace packages/cli`

## Known Limitations

- Release publication is still manual.
- Historical `v0.1.4` notes remain available as the previous release record.
