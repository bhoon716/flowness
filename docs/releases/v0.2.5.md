# Flowness v0.2.5 Release Notes

- Date: `2026-06-21`
- Scope: patch release focused on conversational docs, README/package consistency, review lifecycle changes, and deferrable performance blockers.

## Summary

Flowness 0.2.5 re-centers the project around the conversational workflow harness model: run `flowness init` once, then keep working by natural language with the coding agent. This release also adds Korean and Simplified Chinese READMEs, fixes GitHub/npm README and package metadata links, and makes review output more explicit about finding lifecycle states and hard versus deferrable blockers.

## Added

- `README.ko.md` as a Korean guide.
- `README.zh-CN.md` as a Simplified Chinese guide.
- Release documentation that emphasizes the conversational workflow over command lists.
- Review finding lifecycle statuses and deferrable blocker handling in the release notes and generated templates.
- `docs/troubleshooting/evidence-summary.md` for compact performance evidence review guidance.
- `flowness issue:create --parent-issue` and `--approval-note` for linking follow-up issues with recorded approval text.

## Changed

- The GitHub README and npm README now describe Flowness as a conversational workflow harness instead of a command-heavy CLI.
- `README.md`, `README.ko.md`, `README.zh-CN.md`, and `packages/cli/README.md` now share the same conversational-first positioning and language links.
- `packages/cli/README.md` and `packages/core/README.md` now point to the correct GitHub repository, homepage, and issue tracker.
- Generated AGENTS, navigation, templates, and troubleshooting docs now explain behavior, principles, and usage examples more than command inventories.
- Review output now separates hard blockers from deferrable blockers and records finding lifecycle statuses such as `open`, `addressed`, `closed`, `deferred`, and `accepted-risk`.
- Performance-sensitive changes now expect a compact summary that includes the scenario, baseline, result, workload, key metric, raw report path, limitations, and follow-up issue if any.
- `docs/troubleshooting/performance-improvements.md` now links to the compact evidence summary guidance.
- Follow-up issues can be linked to a parent issue after user approval, and the parent issue can close once only deferrable concerns remain.

## Fixed

- The review gate now allows deferrable performance concerns to move forward when a follow-up issue exists, the finding is marked deferred or accepted-risk, and user approval is recorded.
- Large raw performance evidence no longer fails review by itself when the compact summary is present.
- README/package metadata checks now catch missing Korean and Chinese README links and incorrect package URLs before release.

## Migration Notes

- Existing projects do not need a package rename. Continue using the `@flowness-labs/*` package names.
- If you have performance work in flight, add the compact summary fields to your evidence before asking the review gate to judge it.
- If a review finding is intentionally deferred, create the follow-up issue first and record the approval before committing.

## Breaking Changes

- None.

## Commands

- `npm run build`
- `npm test`
- `npm run audit`
- `npm run release:check`
- `npm pack --dry-run --workspace packages/cli`

## Verification

- Release docs check, build, test, audit, release check, and CLI package dry-run were all part of the release verification.

## Known Limitations

- Deferred performance work still requires a follow-up issue and explicit approval before commit.
- Raw performance artifacts remain useful, but the compact summary is the preferred review surface.
