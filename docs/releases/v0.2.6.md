# Flowness v0.2.6 Release Notes

- Date: `2026-06-22`
- Scope: patch release that publishes the completed hardening work after 0.2.5 was already present on npm.

## Summary

Flowness 0.2.6 packages the hardening work that was completed for the 0.2.5 line under the next publishable patch version. The release keeps the conversational workflow harness positioning, aligns the English/Korean/Chinese README set and npm/GitHub documentation, tightens package metadata, splits the CLI entrypoint without changing commands, and keeps review and commit gates explicit about hard versus deferrable concerns.

## Added

- `README.ko.md` and `README.zh-CN.md` remain part of the release docs set, with cross-links from the English, Korean, and Chinese READMEs.
- `docs/troubleshooting/evidence-summary.md` for compact performance evidence review guidance.
- CLI smoke coverage for the init, run, evidence, review, and commit-gate flow.
- Workflow ID consistency checks so route aliases match the actual workflow blueprints.

## Changed

- Flowness is described as a conversational workflow harness for traceable AI-agent development, not a development operating system.
- README language links stay aligned across English, Korean, and Simplified Chinese.
- Root and npm README positioning stays consistent, including the init-once workflow model.
- Package metadata stays aligned across repository, homepage, and issue tracker fields.
- The CLI entrypoint is split into parser, handlers, services, and formatters while preserving the public command surface.
- Review wording now emphasizes structured and evidence-based review checks instead of implying a deep human-style code reviewer.
- Review and commit gates distinguish hard blockers from deferrable concerns and require follow-up issue and approval handling for the latter.
- Compact evidence summaries are preferred alongside raw artifacts for performance-sensitive review decisions.

## Fixed

- README and package metadata checks now validate the multilingual README links and official GitHub metadata together.
- Commit gating no longer deadlocks on deferrable concerns when the follow-up issue and approval are present.
- The evidence review path accepts large raw artifacts when a compact summary is available.

## Migration Notes

- Existing workspaces can keep using the current `@flowness-labs/*` package names.
- If you have performance-sensitive work in flight, add the compact summary fields before asking the review gate to judge it.
- If a review finding is intentionally deferred, create the follow-up issue first and record approval before commit.

## Breaking Changes

- None.

## Commands

- `npm run build`
- `npm test`
- `npm run audit`
- `npm run release:check`
- `npm pack --dry-run --workspace packages/cli`

## Verification

- Build, test, audit, release check, and CLI package dry-run all passed during release verification.

## Known Limitations

- Deferred performance work still requires a follow-up issue and explicit approval before commit.
- Raw performance artifacts remain useful, but the compact summary is the preferred review surface.
