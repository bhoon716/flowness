# Flowness v0.2.8 Release Notes

- Date: `2026-06-23`
- Scope: narrow release focused on long-session language drift analysis, safer issue allocation, and publish readiness.

## Summary

Flowness 0.2.8 is intentionally narrow. The 0.2.7 workflow consistency fixes already addressed the repeated step/log/review/close loop, so this release does not rework that area unless a regression shows up. Instead, 0.2.8 focuses on two follow-up concerns that showed up in practice: long-running sessions sometimes drift into English in user-facing commentary, and issue creation needs a safer allocation / preview path that does not collide or leak unrelated request labels into issue IDs.

## English Drift Analysis

The drift is not a command surface problem by itself. It usually comes from a combination of:

- long context windows that contain many English technical artifacts,
- internal prompts, workflow files, and command names that are already English-heavy,
- the model falling back to English for progress narration when the session grows long or the surrounding context is mixed-language.

The conclusion is not to ban English. Internal file names, commands, IDs, code symbols, and technical terms should stay English when that is clearer. The lightweight policy added in generated guidance is only for user-facing progress updates and final reports: respond in the user's language when practical, unless the user asks otherwise.

## Added

- Issue creation dry-run / preview output that shows proposed IDs, folder names, titles, slugs, workflows, types, parent links, files, and collision status before files are written.
- Safer issue allocation that derives slugs from the actual issue title and skips occupied numbers instead of reusing collisions.
- A minimal user-facing language policy in generated AGENTS guidance.

## Changed

- Generated AGENTS guidance now tells agents to respond in the user's language unless the user asks otherwise.
- Issue title slugging is now separate from request-intent slugging, so request analysis no longer leaks labels like `LOGIN-PLAN` into persisted issue IDs.
- README and package docs now mention the preview flow and the language policy.
- Version references across the workspace now point to 0.2.8.

## Fixed

- Existing issue workspaces can be reused where practical without rewriting files.
- Issue IDs no longer depend on unrelated request-intent labels when a concrete title exists.
- Multi-issue previews can show proposed allocations before files are created.

## Migration Notes

- Existing workspaces can keep using the current `@flowness-labs/*` package names.
- Use `flowness issue:create --dry-run` or `flowness request:create --dry-run` before approving broader issue creation work.
- User-facing progress updates should follow the user's language when practical; internal IDs, filenames, commands, and technical symbols may remain English.

## Commands

- `npm run build`
- `npm test`
- `npm run audit`
- `npm run release:check`
- `npm pack --dry-run --workspace packages/cli`

## Verification

- Build, test, audit, release check, and CLI package dry-run should pass after the version and release files are updated.

## Known Limitations

- The language policy is lightweight by design; it guides user-facing output without forcing a global language lock.
- Request-intent slugs and title-based issue identifiers are intentionally separate concepts.
